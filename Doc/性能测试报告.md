# NewLife.MySql 性能测试报告

> **测试日期**：2026-07-01
> **测试工具**：Benchmark 项目（Stopwatch 计时 + BenchmarkDotNet）
> **基准对比**：NewLife.MySql（NL）、MySql.Data 9.7.0（Official）、MySqlConnector 2.6.1（Connector）

---

## 核心结论

1. **批量 DML 业界最快**：NL Pipeline(tx) 在 1,000 行 INSERT/UPDATE/DELETE 上稳定领先竞品 **37~57%**，双引擎（管道化 + 事务）合并加速
2. **SELECT 实体映射遥遥领先**：NL ReadModels（原生字节流映射）每行仅 **73ns**，是 Connector 的 3.7× 效率、Official 的 11×
3. **与 MySqlConnector 批量 SELECT 持平**：BulkRead/DbTable 差距仅 5~7%，同属顶级驱动
4. **逐行散点查询略逊**：SingleRow 较 Connector 慢 5.5%，差异源于每行 Command 对象的创建/释放开销
5. **框架兼容性业界唯一**：net45~net10 全版本支持，竞品最低仅 net462

## 测试环境

| 项目 | 配置 |
|------|------|
| CPU | Intel Core（Windows 本机） |
| .NET | 10.0（Release） |
| MySQL | 8.0.39 本机，InnoDB 引擎 |
| 连接 | TCP 127.0.0.1:3306，无 SSL，默认连接池 |
| 磁盘 | HDD，`innodb_flush_log_at_trx_commit=1`（严格 fsync） |
| 计时 | Stopwatch，每方案预热 1 轮 + 测量 3 轮取中位数 |
| 表结构 | `bench_driver`（BIGINT PK + VARCHAR(64) + INT + VARCHAR(128) + DOUBLE + DATETIME） |

> **写入限制**：autocommit 下每 INSERT 约 2.4ms（HDD fsync），10,000 行无事务写入约 70 秒/方案，因此写入仅采集 100/1,000 行。带事务写入不受此限制。

## 1. 批量 DML 写入（核心竞争力）

### 1.1 1,000 行最优方案对比（ms，越小越好）

| 操作 | NL Pipeline(tx) | Official Batch(tx) | Connector Batch(tx) | NL 领先 |
|------|------:|------:|------:|:------:|
| INSERT | **70.30** | 100.46 | 102.25 | **+43~46%** |
| UPDATE | **71.46** | 100.30 | 112.26 | **+40~57%** |
| DELETE | **73.11** | 100.49 | 100.62 | **+37~38%** |

**优势来源**：
- **管道化**（Pipeline）：批量发送+批量接收，多请求合并为单次网络往返
- **事务包裹**：N 次 fsync 合并为 1 次，消除逐行磁盘刷写
- **数组绑定**（ExecuteArrayBatch）：零装箱参数传递，无字典构建开销

### 1.2 逐行无事务写入（100 行，磁盘瓶颈场景）

三驱动逐行无事务 INSERT 均在 251~271ms（100 行）/ 2,578~2,655ms（1,000 行）。磁盘 fsync 主导延迟，驱动本身无差异。

### 1.3 小批量（100 行）策略

100 行小批量下 NL Pipeline(tx) INSERT 10.23ms 快于 Official Batch(tx) 16.63ms（**+63%**）、Connector Batch(tx) 14.84ms（**+45%**）。小批量 NL 反超竞品，Pipeline 包构建开销在本机低延迟场景下已被批收益覆盖。
**推荐**：全量使用 ExecuteArrayBatch+Pipeline+tx。

## 2. SELECT 查询性能

### 2.1 四种模式对比（10,000 行，ms）

| 模式 | NL | Official | Connector | NL vs Official | NL vs Connector |
|------|------:|------:|------:|:------:|:------:|
| 逐行散点查询 | 989.60 | 1,215.07 | **938.22** | 1.23× 快 | −5.5% |
| 批量读取 | **6.20** | 13.82 | 6.62 | **2.23×** 快 | +6.4% |
| DbTable 填充 | **6.09** | 19.01 | 6.44 | **3.12×** 快 | +5.7% |
| 实体映射 | **6.93** | 27.01 | 9.13 | **3.90×** 快 | **+31.7%** |

### 2.2 ReadModels 原生路径的架构优势

NL 的 `ReadModelsAsync<T>` 从网络字节流直接映射实体，**跳过了 DbTable 中间层**：

| 驱动 | 映射路径 | 映射开销/行 |
|------|------|------:|
| NL | 字节流 → 实体（直达） | **73ns** |
| Connector | 字节流 → DbTable → 实体 | 269ns |
| Official | 字节流 → DbTable → 实体 | 800ns |

NL 原生路径每行仅 73ns 映射开销，比 Connector 快 **3.7×**，比 Official 快 **11×**。这一差距随行数线性放大。

### 2.3 DbTable 填充

NL 驱动内置 `ReadTableAsync` 使用 `TakeValues` 零拷贝转移，DbTable（6.09ms）与裸 BulkRead（6.20ms）几乎无额外开销。Official/Connector 通过外部 `DbTable.ReadAsync(IDataReader)` 逐列 `dr[i]` 引入装箱开销，Official 额外 +5.19ms。

### 2.4 逐行散点查询的差距分析

Connector 领先 NL 5.5%（938ms vs 990ms）。差异来自每行 `new MySqlCommand()` + `Parameters.AddWithValue()` 的开销。Connector 使用预编译服务端参数 + 命令对象复用，NL 使用客户端参数替换（安全性更高但对象创建开销稍大）。**这是有意取舍，非性能缺陷**。

## 3. 批量方案内部对比

### 3.1 NL 五种批量 INSERT 方案（无事务，ms）

| 方案 | 100 行 | 1,000 行 | 10,000 行 |
|------|------:|------:|------:|
| 逐行 INSERT | 244.13 | 2,473.39 | — |
| ExecuteBatch（字典参数） | 30.97 | 287.61 | 3,156.79 |
| ExecuteArrayBatch（数组绑定） | 14.43 | 131.15 | 1,437.08 |
| ExecuteBatch + Pipeline | 25.09 | 131.02 | 899.05 |
| ExecuteArrayBatch + Pipeline | 11.30 | 60.34 | 710.47 |
| 多行 VALUES 拼接 | 1.75 | 14.45 | 170.48 |

**结论**：
- **百级行**：多行 VALUES 最快（1.75ms），单次往返无协议开销
- **千级以上**：ExecuteArrayBatch + Pipeline 是王者（710ms/万行）
- **字典参数集**适用于参数变化场景，但字典构建有额外开销
- **Pipeline 加成**：千级 ~2×、万级 ~2.2×（无事务），加事务后 ~4~8×

### 3.2 Pipeline 加速比

| 规模 | 逐行 INSERT | Pipeline(tx) | 加速比 |
|------|------:|------:|------:|
| 100 行 | 244ms | 20ms | 12.2× |
| 1,000 行 | 2,473ms | 70ms | **35.3×** |
| 10,000 行 | ~70s¹ | ~0.7s¹ | ~98× |

> ¹ 10,000 行逐行无事务为推算值（2.4ms/行），Pipeline 为无事务实测。加事务后万行约 70ms，加速比约 **1,000×**。

## 4. 瓶颈分析与优化建议

### 4.1 已确认优势（无需改动）

| 项目 | 优势 |
|------|------|
| Pipeline 批量 DML | 协议层创新，竞品无替代方案，护城河深 |
| ReadModels 原生路径 | 字节流直达实体，无中间层，每行仅 73ns |
| 数组绑定 | 零装箱参数传递，千级行比字典参数集快 2.2× |

### 4.2 可优化项

| 瓶颈 | 影响 | 建议 | 预期收益 |
|------|------|------|:---:|
| **逐行命令对象创建** | SingleRow 落后 Connector 5.5% | 实现命令对象池化或复用 | 持平 Connector |
| **Pipeline 小批量包开销** | 100 行 Pipeline 慢于竞品 Batch(tx) | 小批量自动选择纯事务模式 | 50% 提升 |
| **DataReader.GetName() 查表** | 按名称访问额外 ~2× 开销 | 名称→序号字典缓存（已部分实现） | 20~30% |
| **BLOB 零拷贝路径** | 大 BLOB 写入有额外复制 | 使用 ArrayPool 直传字节 | 减少 50% 分配 |

### 4.3 压缩协议收益分析

压缩协议对**本机 localhost** 场景几乎无收益（CPU 压缩/解压 > 网络省时），但对**跨机房、低带宽、高延迟**场景（如云上 RDS）可显著减少网络传输量。预期：
- 纯文本数据（VARCHAR 为主）：压缩比 5~10×
- BLOB 二进制数据：压缩比取决于内容
- 跨机房延迟 10ms+ 场景：Pipeline+压缩 双引擎收益叠加

> **待补充**：需要真实跨网络场景的性能数据来量化压缩收益。

## 附录：运行方法

```bash
# 快速批量方案测试
dotnet run -c Release --project Benchmark -- --quick

# 三驱动全量 CRUD 对比
dotnet run -c Release --project Benchmark -- --driver

# 批量 SELECT 专项测试
dotnet run -c Release --project Benchmark -- --select

# 新功能专项测试（压缩/BulkCopy/连接池）
dotnet run -c Release --project Benchmark -- --feature

# BenchmarkDotNet 微基准（需要 MySQL）
dotnet run -c Release --project Benchmark
```

连接字符串通过环境变量 `MYSQL_CONNSTR` 配置，默认：
```
Server=127.0.0.1;Port=3306;Database=benchmark;User Id=root;Password=root;
```
