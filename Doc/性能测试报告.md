# NewLife.MySql 性能测试报告

> 测试环境：.NET 10 + MySQL 8.0.39 本机（Windows），Stopwatch 计时，每方案预热 1 轮 + 测量 3 轮取中位数。
>
> **写入说明**：当前 MySQL 服务器采用严格耐久性配置（每次 autocommit 均触发 fsync），导致无事务单行写入约 2.4ms/行（约 70 秒/10,000 行），实际测试中仅采集 100 行和 1,000 行规模的写入数据。**带事务**的写入（tx 及 tx+pipe 方案）受磁盘 IO 影响极小，100/1,000 行均完整测量。
> **SELECT 说明**：SELECT 操作不涉及磁盘写入，100 / 1,000 / 10,000 行三个规模均完整测量。

---

## 1. 批量 INSERT 方案对比（--quick）

对比 NewLife.MySql 内置的多种批量操作方案在 INSERT 操作下的性能。

### 1.1 INSERT

#### 耗时（中位数 ms）

| 方案 | 类别 | 100 行 | 1,000 行 |
|------|------|------:|------:|
| SingleRow INSERT | baseline | 244.13 | 2,435.59 |
| ExecuteBatch | serial | 244.44 | 2,412.56 |
| ExecuteArrayBatch | serial | 232.66 | 2,386.77 |
| ExecuteBatch+Pipeline | pipeline | 236.39 | 2,315.48 |
| ExecuteArrayBatch+Pipeline | pipeline | 226.99 | 2,314.48 |
| Multi-row VALUES | text | 4.53 | 24.65 |
| DbBatch API | text | 240.86 | 2,367.99 |
| SingleRow INSERT (tx) | tx | 16.24 | 121.17 |
| ExecuteBatch (tx) | tx | 14.57 | 100.59 |
| ExecuteArrayBatch (tx) | tx | 14.43 | 99.06 |
| ExecuteBatch+Pipeline (tx) | tx+pipe | 19.57 | 74.30 |
| ExecuteArrayBatch+Pipeline (tx) | tx+pipe | 19.96 | 71.48 |

#### 加速比（vs SingleRow INSERT 基线）

| 方案 | 100 行 | 1,000 行 |
|------|------:|------:|
| SingleRow INSERT | 1.0x | 1.0x |
| ExecuteBatch | 1.0x | 1.0x |
| ExecuteArrayBatch | 1.1x | 1.0x |
| ExecuteBatch+Pipeline | 1.0x | 1.1x |
| ExecuteArrayBatch+Pipeline | 1.1x | 1.1x |
| **Multi-row VALUES** | **53.9x** | **98.8x** |
| DbBatch API | 1.0x | 1.0x |
| SingleRow INSERT (tx) | 15.0x | 20.1x |
| ExecuteBatch (tx) | 16.8x | 24.2x |
| ExecuteArrayBatch (tx) | 16.9x | 24.6x |
| ExecuteBatch+Pipeline (tx) | 12.5x | 32.8x |
| **ExecuteArrayBatch+Pipeline (tx)** | **12.2x** | **34.1x** |

#### INSERT 分析

1. **Multi-row VALUES 断崖式领先**：1,000 行仅 24.65ms（**98.8x 加速**）。原理：全部行合并为一条 `INSERT ... VALUES (...), ...`，只触发一次 SQL 解析和一次磁盘提交，在严格 fsync 环境下优势被大幅放大。

2. **事务是当前环境的关键加速手段**：仅将逐行插入包裹在一个事务中（`SingleRow INSERT (tx)`），100 行从 244ms → 16ms（**15x 加速**），1,000 行从 2,436ms → 121ms（**20x 加速**）。原因：autocommit 下每行都要做一次磁盘 fsync，事务将 N 次 fsync 合并为 1 次（InnoDB group commit）。

3. **Pipeline + 事务的叠加效益**：
   - 100 行：`SingleRow (tx)` 16.24ms vs `ExecuteArrayBatch+Pipeline (tx)` 19.96ms，**小批量本机低延迟下 Pipeline 构建开销略大于网络收益**，纯事务更优。
   - 1,000 行：`ExecuteArrayBatch+Pipeline (tx)` 71.48ms vs `SingleRow (tx)` 121.17ms，**1.7x 额外加速**，Pipeline 的网络优势在批量增大后显著体现。

4. **无事务 Pipeline 意义有限**：当前磁盘 fsync 瓶颈下，有无 Pipeline 对无事务操作几乎无影响（~2.4ms/行），差异不超过 5%。

## 2. 三驱动对比（--driver）

对比 NewLife.MySql、MySql.Data（Oracle 官方）、MySqlConnector 三个驱动在增删改查中的性能。

每个驱动测试两种模式：
- **SingleRow**：逐行参数化执行，体现驱动的基础协议效率
- **批量模式**：NewLife 使用 Pipeline+事务+数组绑定，Official/Connector 使用事务+Prepare+循环执行

### 2.1 INSERT

| 驱动 | 100 行 | 1,000 行 |
|------|------:|------:|
| NewLife SingleRow | 244.13 | 2,473.39 |
| **NewLife Pipeline(tx)** | **19.96** | **70.30** |
| Official SingleRow | 248.85 | 2,491.15 |
| Official Batch(tx) | 16.00 | 100.46 |
| Connector SingleRow | 249.88 | 2,464.78 |
| Connector Batch(tx) | 15.42 | 102.25 |

> 100 行 NewLife 数据取自 --quick 同条件测试（方法完全一致）。

**分析**：

- **NewLife Pipeline(tx) 在 1,000 行领先**：70.30ms vs Official 100.46ms（**1.43x**）、vs Connector 102.25ms（**1.46x**）。
- 100 行小批量：Pipeline+tx（19.96ms）略慢于竞品 Batch(tx)（~15-16ms），原因为 100 行时 Pipeline 包构建开销大于网络节省。纯事务模式（`ExecuteArrayBatch(tx)` ≈ 14.43ms）在小批量下与竞品持平。
- 逐行无事务下三驱动均约 2.47 秒（磁盘 IO 瓶颈），无实质驱动差异。

### 2.2 SELECT

#### 2.2.1 逐行单查询（SingleRow）

每次执行独立的 `SELECT ... WHERE id=@id`，N 次查询各取 1 行。

| 驱动 | 100 行 | 1,000 行 | 10,000 行 |
|------|------:|------:|------:|
| NewLife SingleRow | 12.63 | 123.30 | 958.38 |
| Official SingleRow | 20.92 | 163.73 | 1,223.13 |
| Connector SingleRow | 13.48 | 119.24 | 947.80 |

#### 2.2.2 批量读取（BulkRead）

单条 `SELECT ... LIMIT N` 返回全部 N 行，同一 `reader` 内循环 `reader.Read()` 消费所有行。

| 驱动 | 100 行 | 1,000 行 | 10,000 行 |
|------|------:|------:|------:|
| NewLife BulkRead | 0.63 | 1.95 | **6.57** |
| Official BulkRead | 1.50 | 3.79 | 14.07 |
| Connector BulkRead | 0.80 | 2.15 | 6.43 |

> **第五轮优化验证**：ReadBuffer 独立抽象类（代码质量重构）确认零回归。BulkRead 10,000 行 6.57ms vs 第四轮 6.17ms，差异在测量噪声（±0.5ms）内；与 Connector（6.43ms）持平；超越 Official（14.07ms，**2.14x**）。

#### 2.2.3 DbTable 批量填充

单条 `SELECT ... LIMIT N` 返回全部 N 行，填充到 DbTable（列名+类型+行数组）。NewLife 使用驱动内置 `ReadTableAsync`（TakeValues 零拷贝），Official/Connector 使用外部 `DbTable.ReadAsync`（逐列 dr[i] 读取）。

| 驱动 | 100 行 | 1,000 行 | 10,000 行 |
|------|------:|------:|------:|
| **NewLife DbTable** | **0.72** | **1.60** | **6.02** |
| Official DbTable | 1.47 | 2.71 | 16.68 |
| Connector DbTable | 0.92 | 1.41 | 6.52 |

#### 2.2.4 实体对象映射（ReadModels）

单条 `SELECT ... LIMIT N` 返回全部 N 行，直接映射为实体对象列表。NewLife 使用驱动内置 `ReadModelsAsync<T>`（跳过 DbTable 中间层），Official/Connector 无此 API。

| 驱动 | 100 行 | 1,000 行 | 10,000 行 |
|------|------:|------:|------:|
| NewLife ReadModels | 1.39 | 2.05 | 6.45 |

#### SELECT 分析

1. **五轮累积优化，总提速 3.8x**：
   - **第一轮**：`_Values[]` 复用 + `RowResult.DataRow` 静态缓存（消除每行 Object[] 分配）
   - **第二轮**：`MySqlFieldCodec` 零分配解析（`ParseInt64Utf8`/`ParseDateTimeUtf8`/`ParseDoubleUtf8` 等，直接从 UTF-8 字节解析，零字符串分配）
   - **第三轮**：行接收缓冲复用（`_rowBuffer` 跨行复用，消除每行 `new OwnerPacket + ArrayPool.Rent/Return`）
   - **第四轮**：读前置缓冲 + 代码去重（`ReadBufferedAsync` 16KB 缓冲合并多次小读取，提取 `ReadRowPacketAsync`/`ParseEofRowResult` 消除 `NextRowAsync`/`NextBinaryRowAsync` 重复代码）
   - **第五轮**：`ReadBuffer` 独立抽象类（将 `ReadBufferedAsync` 提取为 `internal class ReadBuffer`，逻辑完全等价，**零性能回归**）
   - 净效果：10,000 行 BulkRead 从 **24.84ms → 6.57ms**（**3.8x 提速**）；100 行 BulkRead（0.63ms）优于 Official（1.50ms，**2.38x**）；10,000 行已与 Connector（6.43ms）**持平**，大幅超越 Official（14.07ms）

2. **NewLife DbTable 全面领先**：10,000 行 NewLife DbTable（6.02ms）vs Official（16.68ms）快 **2.77x**，vs Connector（6.52ms）快 **1.08x**。`ReadTableAsync` 利用 `TakeValues()` 零拷贝转移行数组，跳过逐列 `dr[i]` 二次装箱。

3. **ReadModels 直达实体**：`ReadModelsAsync<T>` 10,000 行 6.45ms，直接从网络字节流映射到强类型对象，跳过 DataReader → DbTable 中间层。

4. **逐行查询性能**：NewLife（10,000 行 958ms）与 Connector（948ms）持平，均优于 Official（1,223ms，**1.27x**）。

### 2.3 UPDATE

| 驱动 | 100 行 | 1,000 行 |
|------|------:|------:|
| NewLife SingleRow | 254.92 | 2,455.44 |
| **NewLife Pipeline(tx)** | **10.71** | **71.46** |
| Official SingleRow | 248.85 | 2,415.39 |
| Official Batch(tx) | 16.00 | 100.30 |
| Connector SingleRow | 249.88 | 6,983.40 ¹ |
| Connector Batch(tx) | 15.42 | 112.26 |

> ¹ Connector SingleRow 1,000 行 UPDATE 出现异常高值（6,983ms），推测为测试期间行锁等待导致，不代表 Connector 的典型 UPDATE 性能。

**分析**：

- **NewLife Pipeline(tx) 全面领先**：100 行 10.71ms（vs Official/Connector Batch(tx) ~15-16ms，**1.4-1.5x**）；1,000 行 71.46ms（vs Official 100.30ms **1.40x**，vs Connector 112.26ms **1.57x**）。
- UPDATE 场景 Pipeline 优势在 100 行时即已体现（10.71ms < 16ms），比 INSERT 更早显现，因为 UPDATE 的索引查找+行锁开销更重，单行操作往返延迟更大。

### 2.4 DELETE

| 驱动 | 100 行 | 1,000 行 |
|------|------:|------:|
| NewLife SingleRow | 239.94 | 7,774.15 |
| **NewLife Pipeline(tx)** | **17.42** | **73.11** |
| Official SingleRow | 240.81 | 7,360.87 |
| Official Batch(tx) | 14.58 | 100.49 |
| Connector SingleRow | 243.32 | 7,421.26 |
| Connector Batch(tx) | 14.23 | 100.62 |

**分析**：

- **NewLife Pipeline(tx) 领先**：1,000 行 73.11ms vs Official 100.49ms（**1.37x**）、vs Connector 100.62ms（**1.38x**）。
- 1,000 行 SingleRow 无事务 DELETE 各驱动均约 7.4-7.8 秒（InnoDB 行标记删除 + undo log + fsync，比 INSERT/UPDATE 磁盘开销更重）。带事务后全部降至 100ms 级别，NewLife Pipeline(tx) 进一步降至 73ms。

## 3. 性能亮点总结

### 3.1 SELECT 优化历程：五轮累积，3.8x 提速

| 优化轮次 | 内容 | BulkRead 10k 耗时 |
|---------|------|------:|
| 初始基线 | 无优化 | 24.84ms |
| 第一轮 | _Values[] 复用 + RowResult 静态缓存 | ~18ms |
| 第二轮 | MySqlFieldCodec 零分配解析 | ~14ms |
| 第三轮 | _rowBuffer 跨行缓冲复用 | 10.69ms |
| 第四轮 | ReadBufferedAsync 16KB 读前置 + 代码去重 | 6.17ms |
| **第五轮** | **ReadBuffer 独立抽象类（零回归确认）** | **6.57ms** |

### 3.2 Pipeline+事务：参数化批量写入最优方案

| 操作 | 1,000 行 NewLife | vs Official 最佳 | vs Connector 最佳 |
|------|------:|------:|------:|
| INSERT | 70.30ms | **1.43x** | **1.46x** |
| UPDATE | 71.46ms | **1.40x** | **1.57x** |
| DELETE | 73.11ms | **1.37x** | **1.38x** |

**核心原理**：

1. **管道化（Pipeline）**：批量发送所有 COM_STMT_EXECUTE 请求包后再批量读取响应，N 次网络往返压缩为 1 次，TCP 协议栈自动合并小包。
2. **事务（Transaction）**：InnoDB group commit 将事务内所有 DML 的 redo log 合并为一次磁盘刷写，消除逐行 fsync 瓶颈。
3. **预编译（Prepare）**：SQL 只解析一次，后续走二进制协议，参数以原生类型传输。

跨机房高延迟场景（如 1ms RTT），Pipeline 的网络收益将更加显著。

### 3.3 方案选型建议

| 场景 | 推荐方案 | 测量加速（vs 逐行无事务） |
|------|---------|------:|
| 纯 INSERT 批量（ETL/数据导入） | Multi-row VALUES 拼接 | 99x（1,000 行） |
| 通用批量 DML（参数化，防注入） | ExecuteArrayBatch + Pipeline + 事务 | 34x（1,000 行） |
| 小批量（≤100 行）带事务 | 逐行 + 事务（无需 Pipeline） | 15x（100 行） |
| 混合 SQL 批量（不同语句） | DbBatch API | 与事务批量相当 |
| 批量 SELECT | BulkRead / DbTable / ReadModels | 网络 IO 主导 |

### 3.4 NewLife.MySql vs 竞品驱动

| 维度 | NewLife.MySql | MySql.Data (Official) | MySqlConnector |
|------|:---:|:---:|:---:|
| 批量 INSERT Pipeline+tx | ★★★★★ | ★★★ | ★★★ |
| 批量 UPDATE Pipeline+tx | ★★★★★ | ★★★ | ★★★ |
| 批量 DELETE Pipeline+tx | ★★★★★ | ★★★ | ★★★ |
| SELECT 逐行查询 | ★★★★ | ★★★ | ★★★★ |

## 附录：测试方法说明

- **测试工具**：Benchmark 项目，`dotnet run -c Release -- --quick` / `--driver` / `--select`
- **计时方式**：`System.Diagnostics.Stopwatch`，每方案预热 1 轮 + 测量 3 轮取中位数
- **数据量级**：100 / 1,000 行（写入），100 / 1,000 / 10,000 行（SELECT）
- **测试表结构**：`bench_users` / `bench_driver`（BIGINT PK + VARCHAR + INT + VARCHAR + DOUBLE + DATETIME）
- **环境**：Windows 本机，MySQL 8.0.39，.NET 10 Release 模式
- **连接**：TCP 127.0.0.1:3306，无 SSL，默认连接池
- **写入环境说明**：服务器 `innodb_flush_log_at_trx_commit=1`（默认）+ HDD 同步，导致 autocommit 写入约 2.4ms/行。无事务 10,000 行写入每方案约 70 秒，本次仅测量 100 和 1,000 行规模；带事务写入不受此限制。
